# Kenny-Wiki 的個人知識庫 - Wiki Schema

## Wiki 主題
這是 Kenny-Wiki 的個人知識庫，主題聚焦於：個人學習與知識整理

## 使用工具
- AI 工具：Claude Code
- 知識庫語言：繁體中文

## 資料夾結構
- raw/    → 原始資料（只讀，不修改）
- wiki/   → AI 維護的 Wiki 文章
- wiki/index.md → 總索引（每次更新都要同步）

## 文章格式規範
每篇 Wiki 文章必須包含以下區塊（按順序）：

```
# 概念名稱

## 定義
一句話說明這個概念是什麼。

## 重點整理
- 要點一
- 要點二
- 要點三

## 延伸閱讀
- [[相關概念A]]
- [[相關概念B]]

## 來源
- raw/對應檔案名稱
```

## 知識庫分類架構
- 工具使用
- 學習筆記
- 參考資料

## 維護規則
1. 新增文章時，必須同步更新 wiki/index.md
2. 文章之間使用 [[概念名稱]] 格式建立交叉連結
3. 每篇文章結尾必須標註 raw/ 中的來源檔案
4. 禁止刪除或修改 raw/ 資料夾中的任何檔案
5. 文章語言統一使用繁體中文

## Wiki 操作指令
- ingest → 讀取 raw/ 新資料，更新 Wiki
- query [問題] → 從 Wiki 查詢並回答
- lint → 檢查 Wiki 的格式和連結是否正確

## 特別說明
每次執行 ingest 後，請確認以下事項：
- index.md 已更新
- 新文章已加入適當的 [[交叉連結]]
- 沒有孤立的文章（每篇文章至少被一個其他文章連結）

# Karpathy 編程規則（⭐ 19.5K — Andrej Karpathy 開發哲學）

> 資料來源：https://github.com/forrestchang/andrej-karpathy-skills

## 原則一：先想再寫（Think Before Coding）
- 遇到模糊需求，先明確說出假設，提出問題，不可靜默腦補
- 沒弄清楚邏輯與需求前，一行程式碼都不准動
- 要說明你打算怎麼做，再動手

## 原則二：極簡主義（Simplicity First）
- 只實作被要求的功能，不准加沒被要求的東西
- 能用 10 行解決的，不准寫 100 行
- 拒絕過度抽象與過度工程
- 沒有被要求的功能，一個字都不要多寫

## 原則三：外科手術式修改（Surgical Changes）
- 只修改與需求直接相關的程式碼
- 不准亂動我的註解、格式或沒壞掉的程式碼
- 不要「順便優化」或重構沒被提到的部分
- 保留既有的程式風格

## 原則四：目標驅動執行（Goal-Driven Execution）
- 給你「成功標準」或測試案例，你要自己跑 Loop 修正直到通過
- 不要等我一步一步下指令，給你目標，你來達成
- 「LLM 擅長持續循環直到達成目標。不要告訴它做什麼，給它成功標準，然後看它執行。」— Andrej Karpathy
